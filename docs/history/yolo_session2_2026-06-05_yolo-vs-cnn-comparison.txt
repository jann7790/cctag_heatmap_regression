```
Tiles: 6x3        ← 4096×2160 用 1024 視窗 + 30% 重疊,橫切 6 塊、縱切 3 塊 = 18 塊

after NMS: 1      ← IoU 合併重複後剩 1 個(就是那個 marker)
```

- **Heatmap 之所以 258 FPS,是因為它把 4K 縮成 640×400 丟掉了細節** —— 跟「YOLO 整張縮 1024」那行(81 FPS)是同一種妥協,40m 的小 marker 在這個解析度下一樣會不見。

> 不是「heatmap 快、YOLO 慢」。是 **「整張縮圖(快、丟小物件)」vs「切 tile(慢、保留小物件)」**。這個選擇兩種架構都得面對。你的需求是「保留 cctag px」,那無論用哪個模型,都得付切 tile 的時間代價;heatmap 唯一的優勢是同樣 tile 數下,每塊 forward 比 YOLO 略省一點。

## 3. ⚠️ 重要反轉:GPU 和 CPU 的勝負相反

```
GPU @1024 單塊:  Heatmap 6.4ms  <  YOLO 12.4ms   → heatmap 快
CPU @1024 單塊:  Heatmap 277ms  >  YOLO 102ms    → YOLO 快 2.7×
```

為什麼會翻盤,而且這也修正了我之前的說法:

1. **我之前說「heatmap GPU 快 2×」其實有水分** —— YOLO 那 12.4ms 含 NMS + Python 前後處理 overhead(純 forward 只 ~2ms)。在 GPU 上模型本體 YOLO 反而更輕。
2. **CPU 是 FLOPs-bound,藏不住**:resnet18 在 1024×1024 的計算量(~38 GFLOPs + 上採樣 decoder)其實**比 yolo11n(~16 GFLOPs)大**。CPU 沒有 GPU 的平行度去掩蓋,所以 heatmap 真實的「比較重」就暴露了 → 慢 2.7×。

→ 結論:**heatmap 不是無條件比較快**。它在 GPU + 小輸入(640)時快;一旦上 1024 tile、或跑 CPU,它的 decoder 反而變包袱,YOLO 更划算。

## 4. 兩者 CPU 都不可用於 4K tiled
0.2~0.5 FPS,一張要 2~5 秒。4K 切 tile 這條路**必須 GPU**。CPU 只有「整張縮 640 跑 heatmap」(65ms,15 FPS)勉強能用,但那就丟掉小 marker 了。

## YOLO 的好處(相對 CNN heatmap)

1. **給 bounding box,不只中心點** —— YOLO 直接輸出 marker 的範圍/大小;這個 heatmap 只給一個峰值點。要做距離估計、ROI 裁切、pose,box 尺寸很有用。
2. **天生多目標** —— 一張圖任意數量的 marker,各自有信心分數。heatmap 單峰 decode 一個 tile 只給一個峰,多 marker 要自己寫多峰。
3. **近距/大 marker 更穩** —— 數據показ:5m、10m 大 marker CNN 漏了,YOLO 抓到。檢測頭天生處理大範圍物件。
4. **成熟生態** —— ultralytics 一套搞定訓練/增強/匯出(ONNX/TRT/CoreML)/追蹤/mAP 評估;還能用 COCO 預訓練 transfer learning。CNN 這邊是手寫腳本,什麼都得自己來。
5. **每框獨立信心分數 + 可多類別** —— 之後要分不同 marker 種類,YOLO 直接擴。
6. **標準評估指標(mAP)**,好調好比較。

## 但反過來 CNN heatmap 的好處(平衡一下)

- **中心定位更精準**(subpixel 峰值)—— 對 CCTag 這種要精確圓心做三角定位/pose 的場景,heatmap 的中心比 YOLO 的 box 中心準。
- **假框極少**(precision 高),暗圖環境乾淨。
- **更快**(前面測過)。

## Q1:「看到夠多圈就咬框」會不會 mis-predict?

**會,而且這正是 YOLO 假框的來源** —— 同一個機制的一體兩面:

```
看到部分圈就敢咬  →  優點:大marker/被切/部分遮擋 也抓得到(高 recall)
                  →  缺點:暗圖的光暈、同心反光、圓形雜訊 也咬(低 precision)
```

這跟我們的數據完全對得上:**YOLO 在暗光圈圖噴 7~10 個假框**,就是它把「夠像圈的東西」當 marker 了。而且部分咬還會**框歪**(只看到半個 marker 的 tile,框的中心/大小都是錯的)—— 5m 那兩個碎框就是。

反過來 **CNN 乾淨(precision 高)也是同一個原因**:它要求「看到完整、置中的同心圖案」才會出峰值,門檻嚴 → 假框少,但代價就是 marker 一大/一缺就漏。

> 所以「YOLO 高 recall 低 precision」vs「CNN 高 precision 低 recall」不是兩個模型好壞,是**同一條 precision/recall 取捨線上的兩個點**。沒有免費的午餐。

## Q2:YOLO 能做的,CNN 也能吧?

**架構上幾乎都能 —— 因為這個 heatmap 模型本質就是 CenterNet,一種偵測器**:

| YOLO 的能力 | CNN heatmap 能不能 | 怎麼做 |
|---|---|---|
| 輸出 bounding box | ✅ 能 | 模型已經有 **size head**(輸出 ellipse_a/b)+ **offset head** → 峰值 + 尺寸 = box |
| 多目標 | ✅ 能 | 現在 decode 只取 argmax 1 個峰;改 **multi-peak NMS** 就多目標 |
| 每個偵測有信心 | ✅ 能 | 峰值高度就是信心 |
| 多類別 | ✅ 能 | 多通道 heatmap |

**但關鍵在這**:就算讓 CNN 也輸出 box、也多目標,它**不會自動獲得 YOLO「部分就咬」的高 recall** —— 那個高 recall 是 YOLO 的訓練目標(grid/anchor 寬鬆回歸)帶來的,**你想讓 CNN 也那樣寬鬆,它就會跟著長出 YOLO 那些假框**。

> 一句話:**CNN 能做 YOLO 的「輸出格式」(box、多目標、信心),但「寬鬆咬 vs 嚴格咬」是訓練目標決定的取捨,換到哪個模型都得付一樣的 precision/recall 代價。**

## 對你的實際意義
- 對 CCTag 這種要**精確圓心**的 fiducial,CNN 的「嚴格、置中、subpixel」其實是**優點**,不是缺點 —— 它天生就在抓你真正要的圓心。
- YOLO 的價值在**先粗抓不漏**(acquisition),適合當第一階段;CNN 適合第二階段**精定位**。這也正是你 repo 註解裡寫的 "two-stage localization"(YOLO 粗抓 → CNN 精修)的道理。

## 為什麼 20px:沒有 offset head + 我把 sigma 調大

| 因素 | 影響圓心精度 |
|------|------|
| **沒開 offset head** ❌ | 圓心只能落在 heatmap 的 **stride-4 格點**(4px 量化),靠 argmax+拋物線硬湊 → 這是 20px 的主因 |
| **sigma 3.0(我調大的)** | 峰更糊 → 更難精準定位。我為了「等比」調大,反而**傷了精度** |
| stride 4 | 4px 量化網格 |
| 還沒收斂 + 難樣本拉高均值 | 20px 是 mean,中位數應低很多,但仍受前兩項拖累 |

**offset head 的作用**就是 CenterNet 那招:除了峰值,再回歸一個**次像素 (dx,dy) 偏移** → 圓心精度從「4px 量化」直接拉到**次像素級**。漏開它,精度當然崩。

「去重半徑」是我推理流程裡的一個參數,解釋給你:

## 為什麼需要「去重」

我把 4K 圖切成**重疊的 tile**(overlap 30%)。所以一個 marker 如果剛好坐在兩塊 tile 的重疊區,**會被好幾塊 tile 各自偵測一次** → 同一個實體 marker 冒出**好幾個峰值**,座標差一點點。

```
   tile A ┐
          ├─ 重疊區有個 marker
   tile B ┘   → A 測到一個峰、B 也測到一個峰 → 同一marker 變2個
```

## 「去重半徑」做什麼

把**距離很近的峰值視為同一個 marker、只留一個**(分數最高的)。那個「多近算同一個」的距離門檻,就是**去重半徑**。

我現在寫死 **25px**:兩個峰若相距 <25px → 當成同一 marker 合併;>25px → 當成兩個。

```python
if 兩峰距離 > 25px:  # ← 這個 25 就是去重半徑
    當成不同 marker
```

## 為什麼現在出問題

**25px 對大 marker 太小**。5m 的 marker 直徑 ~1000px,兩塊 tile 對它圓心的估計可能差 30-50px(marker 很大,估計本來就有點抖)→ 距離 >25px → 沒被合併 → **同一個大 marker 顯示成 2 個十字**(galvo_40m 那張就是)。

## 怎麼修

讓去重半徑**跟著 marker 大小縮放**(用 size head 預測的半徑),而不是固定 25px:
```
去重半徑 = marker 半徑 × 0.5   # 大marker用大半徑、小marker用小半徑
```
這樣不管 marker 多大,「大約一個 marker 範圍內」的重複峰都會被併成一個。
